# 乐高故事书籍功能 - 执行规范要求

本文档定义了项目开发过程中必须遵守的所有规范要求，每个阶段的"执行规范检查"任务必须检查本文件中的所有内容。

---

## 一、测试执行规范

### 1.1 测试执行原则
1. **每个阶段代码生成后立即测试**: 单元测试 → 功能测试 → API测试
2. **断言验证内容**: 测试断言必须验证返回内容符合预期，不能只是函数调用断言
3. **函数覆盖率要求**: 所有测试必须生成真实执行后的覆盖率报告(行覆盖率)，覆盖率 > 90%
   - **测试文件必须导入源代码**: 测试文件不能重新定义函数
   - **覆盖率检查命令**: 每个阶段必须执行 `npm run test:coverage` 命令
   - **覆盖率阈值**: vitest 配置已设置 90% 阈值，低于阈值测试将失败
   - **覆盖率报告**: 覆盖率报告生成在 `coverage/` 目录
4. **子系统联调测试**: 子系统完成后进行联调测试，使用本地 SQLite 数据库
5. **端到端测试**: 页面完成后进行 E2E 测试，模拟真实用户操作，环境与线上一致
6. **测试必须完全成功**: 每个阶段的所有测试必须100%通过，不允许有skip的测试用例，如有失败的测试必须修复后才能进入下一阶段

### 1.2 测试环境要求
- **单元测试**: 使用 Mock 隔离外部依赖
- **API测试**: 连接本地 SQLite 数据库
- **联调测试**: 使用本地 SQLite 数据库，模拟真实数据流
- **E2E测试**: 使用 Playwright/Puppeteer，启动本地服务器，环境与线上基本一致

### 1.3 覆盖率检查要求
- [ ] 执行 `npm run test:coverage` 检查覆盖率
- [ ] 验证语句覆盖率 >= 90%
- [ ] 验证函数覆盖率 >= 90%
- [ ] 验证分支覆盖率 >= 90%
- [ ] **验证行覆盖率 >= 95%（必须要求）**
- [ ] 如果覆盖率不足，补充测试用例直到达标
- [ ] 覆盖率报告生成在 `coverage/` 目录

## 二、静态代码检查规范

### 2.1 静态检查执行原则
1. **每个阶段代码生成后立即检查**: 在测试之前执行静态代码检查
2. **高内聚低耦合检查**: 验证模块职责单一、依赖关系清晰
3. **潜在 Bug 检测**: 检测空指针、未定义变量、类型错误、资源泄漏等
4. **需求符合性检查**: 验证代码实现符合需求文档要求
5. **设计规范符合性检查**: 验证代码符合设计文档规范

### 2.2 静态检查工具配置
- **ESLint**: JavaScript 代码规范检查
- **Stylelint**: CSS 样式规范检查
- **HTMLHint**: HTML 代码规范检查
- **自定义检查脚本**: 高内聚低耦合、需求符合性检查

### 2.3 静态检查内容清单

#### 代码质量检查
- [ ] 代码风格一致性检查
- [ ] 命名规范检查（变量、函数、类）
- [ ] 注释完整性检查
- [ ] 代码格式化检查

#### 高内聚低耦合检查
- [ ] 模块职责单一性检查（每个模块只负责一个功能）
- [ ] 函数长度检查（单个函数不超过 50 行）
- [ ] 文件长度检查（单个文件不超过 800 行）
- [ ] 依赖关系检查（避免循环依赖）
- [ ] 接口隔离检查（接口最小化）

#### 潜在 Bug 检测
- [ ] 空值/未定义检查（null/undefined 访问）
- [ ] 类型错误检查（隐式类型转换）
- [ ] 数组越界检查
- [ ] 资源泄漏检查（未关闭的连接、文件句柄）
- [ ] 异步错误处理检查（未捕获的 Promise 异常）
- [ ] 内存泄漏检查（未清理的事件监听器、定时器）

#### 需求符合性检查
- [ ] 功能完整性检查（所有需求功能已实现）
- [ ] 参数验证检查（输入参数边界验证）
- [ ] 错误处理检查（异常情况处理）
- [ ] 安全性检查（XSS、CSRF、SQL 注入防护）

#### 设计规范符合性检查
- [ ] API 接口规范检查（RESTful 规范）
- [ ] 数据库操作规范检查（参数化查询）
- [ ] 前端组件规范检查（乐高风格一致性）
- [ ] 日志规范检查（敏感信息脱敏）

## 三、部署规范检查要求

### 3.1 部署环境约束
- **部署平台**: 仅限 Cloudflare Pages（不允许使用 Cloudflare Workers）
- **后端逻辑**: 仅限 Cloudflare Page Functions
- **数据库**: 仅限 Cloudflare D1
- **路由**: 仅限静态路由（不允许动态路由）
- **代码托管**: GitHub（自动部署到 Cloudflare Pages）

### 3.2 部署规范检查内容

#### Cloudflare Pages 规范检查
- [ ] 项目结构符合 Pages 要求（静态文件在根目录或 public 目录）
- [ ] functions 目录结构正确（functions/api/*.js）
- [ ] 无 Workers 特有 API 使用（如 addEventListener）
- [ ] 无服务端渲染框架使用（如 Next.js SSR）
- [ ] 静态资源路径正确（相对路径或绝对路径）

#### Page Functions 规范检查
- [ ] 函数导出格式正确（export async function onRequest*）
- [ ] 使用标准 Request/Response 对象
- [ ] 使用 context.env 获取环境变量
- [ ] 使用 context.request 获取请求对象
- [ ] 无 Node.js 特有模块使用（如 fs, path）
- [ ] 无动态路由参数（如 /api/:id）

#### D1 数据库规范检查
- [ ] SQL 查询使用参数化（防止 SQL 注入）
- [ ] 使用 context.env.DB 执行查询
- [ ] 无本地文件数据库操作
- [ ] 无其他数据库连接（如 MySQL, PostgreSQL）
- [ ] 数据库操作在 Page Function 中执行

#### 静态路由规范检查
- [ ] 所有页面为静态 HTML 文件
- [ ] 无服务端路由重写
- [ ] 无动态路由参数
- [ ] URL 参数通过查询字符串传递

#### 环境变量规范检查
- [ ] 敏感信息通过环境变量配置
- [ ] 环境变量在 wrangler.toml 中声明
- [ ] 无硬编码的 API Key
- [ ] 无硬编码的数据库连接信息

### 3.3 部署规范检查执行时机
- **每个阶段代码生成后**: 在静态代码检查之后、测试之前执行
- **发现违规立即修复**: 不允许积累到部署阶段

## 四、接口一致性检查要求

### 4.1 前后端接口一致性
- [ ] 检查前端API调用与后端接口路径是否一致
- [ ] 检查前端请求参数与后端接收参数是否一致
- [ ] 检查后端返回数据与前端处理逻辑是否一致
- [ ] 检查错误码和错误消息是否一致

### 4.2 数据库字段一致性
- [ ] 检查数据库字段与Schema定义是否一致
- [ ] 检查API返回字段与数据库字段是否一致
- [ ] 检查字段类型是否正确
- [ ] 检查字段约束是否正确

### 4.3 函数调用一致性
- [ ] 检查函数参数与返回值类型是否一致
- [ ] 检查函数调用时参数顺序是否正确
- [ ] 检查异步函数是否正确处理

---

## 五、代码修改后强制检查要求

### 5.1 每次代码修改后必须执行
```
npm run lint          # 静态代码检查
node scripts/check-deployment.js  # 部署规范检查
npm run test:coverage # 所有测试（含覆盖率）
```

### 5.2 检查通过要求
- [ ] 静态代码检查通过（0 错误，警告可接受）
- [ ] 部署规范检查通过
- [ ] 所有测试通过
- [ ] 覆盖率 >= 90%
- [ ] 三个检查全部通过后才能继续执行下一个任务

---

## 六、Bug 记录要求

### 6.1 Bug 记录原则
1. **每个阶段记录所有 Bug**: 包括发现、修复、验证全过程
2. **详细记录原因和解决方案**: 用于反思分析
3. **分类统计 Bug 类型**: 用于识别薄弱环节
4. **关联到具体代码位置**: 便于追溯

### 6.2 Bug 记录格式
每次检查发现的问题必须记录到 `spec/bug_records.md` 文件中，包括：
- [ ] 发现时间
- [ ] 发现环节（静态检查/部署检查/单元测试/API测试/联调测试）
- [ ] 问题描述
- [ ] 代码位置
- [ ] 严重程度
- [ ] 解决方案
- [ ] 验证结果

---

## 七、阶段反思要求

### 7.1 反思执行原则
1. **每个阶段完成后立即反思**: 在所有测试通过后进行阶段反思
2. **苏格拉底式提问方法**: 通过提问引导发现问题的根本原因
3. **记录问题和解决方案**: 将问题、原因、解决方案记录到反思文档
4. **更新流程规范**: 根据反思结果更新 task_list 和 check_list
5. **下一阶段应用改进**: 确保改进措施在下一阶段得到执行

### 7.2 反思文档要求
- [ ] 产出阶段反思文档
- [ ] 使用苏格拉底式提问分析问题
- [ ] 记录发现的问题和解决方案
- [ ] 更新规范文档

---

## 八、需求反思要求

### 8.1 需求反思执行原则
1. **每个阶段完成后立即反思需求**: 在阶段反思之后进行需求反思
2. **苏格拉底式提问验证需求**: 通过提问验证需求是否被正确满足
3. **用户视角验证**: 从用户使用角度验证功能是否合理、顺畅
4. **儿童友好性验证**: 验证功能是否符合10-12岁儿童使用习惯
5. **更新需求文档**: 如发现问题需更新需求文档，并同步更新 task_list 和 check_list，确保改进措施在下一阶段得到执行

### 8.2 需求反思文档要求
- [ ] 验证需求是否完整实现
- [ ] 检查是否有遗漏的需求点
- [ ] 验证用户使用流程是否顺畅
- [ ] 验证是否符合儿童使用习惯
- [ ] 输出需求反思文档

---

## 九、测试需求文档要求

### 9.1 测试需求文档内容
每个阶段完成后，必须产出测试需求文档，包括：
- [ ] 功能测试需求
- [ ] 边界测试需求
- [ ] 异常测试需求
- [ ] 性能测试需求（如适用）
- [ ] 安全测试需求（如适用）

### 9.2 测试需求反思要求
产出测试需求文档后，使用苏格拉底式提问反思测试需求是否完整：
- [ ] 是否覆盖了所有功能点？
- [ ] 是否考虑了边界条件？
- [ ] 是否考虑了异常情况？
- [ ] 是否考虑了用户操作习惯？
- [ ] 是否考虑了儿童使用场景？
- [ ] 现有测试用例是否足够？
- [ ] 是否需要补充测试用例？
1. **反思和改进**: 确保测试需求文档的测试需求全部生成测试用例，并全部执行通过
