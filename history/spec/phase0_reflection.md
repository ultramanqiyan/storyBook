# 阶段0反思文档 - 环境准备与项目初始化

## 一、苏格拉底式提问分析

### 1.1 关于项目结构
**问题**: 为什么需要这样的目录结构？
**回答**: 
- `functions/api/` - Cloudflare Pages Functions 要求的目录结构
- `tests/` - 测试文件独立目录，便于管理和排除覆盖率
- `migrations/` - 数据库迁移脚本，便于版本控制
- `css/js/assets/` - 前端静态资源分离，便于维护

### 1.2 关于测试覆盖率
**问题**: 为什么测试覆盖率要求这么高（行覆盖率95%，函数覆盖率90%）？
**回答**: 
- 高覆盖率可以确保代码质量
- 减少生产环境的bug
- 便于后续重构和维护
- 符合需求文档的强制要求

### 1.3 关于部署规范
**问题**: 为什么只能使用 Cloudflare Pages 而不是 Workers？
**回答**: 
- 需求文档明确要求只能部署到 Cloudflare Pages
- Pages 适合静态网站和简单的 API
- Workers 适合更复杂的后端逻辑
- 遵循需求约束是项目成功的关键

## 二、发现的问题和解决方案

### 2.1 ESLint 配置问题
**问题**: 测试文件中的 describe 函数超过50行限制
**解决方案**: 在 ESLint 配置中添加 overrides，对测试文件禁用 max-lines-per-function 规则

### 2.2 覆盖率配置问题
**问题**: 配置文件被计入覆盖率统计
**解决方案**: 在 vitest.config.js 的 exclude 中添加配置文件排除规则

### 2.3 部署检查脚本问题
**问题**: 检查脚本中的行过长
**解决方案**: 将数组拆分为多行，保持每行不超过120字符
