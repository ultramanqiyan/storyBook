## 〇、产品介绍（非技术视角）

### 产品是什么？

卡牌互动小说网站是一个让用户通过收集卡牌、选择角色和情节来创作和阅读互动小说故事的在线平台。

### 核心玩法

| 玩法 | 说明 |
|------|------|
| **创建书籍** | 选择书籍类型（儿童冒险/魔幻传说/都市言情/职场风云） |
| **创建角色** | 设计主角和配角，设定性格和关系 |
| **收集卡牌** | 通过解谜获得天气、地形、装备等卡牌 |
| **生成章节** | 选择卡牌组合，AI辅助生成故事章节 |
| **解谜互动** | 每个章节包含谜题，增加互动性 |

### 目标用户

10-40岁，喜欢故事创作和卡牌收集的用户，主要面向美国、欧洲等海外市场。

### 商业价值

| 价值点 | 说明 |
|--------|------|
| 免费增值模式 | 基础功能免费，高级卡牌付费 |
| 用户留存 | 卡牌收集机制增加用户粘性 |
| 内容生态 | 用户创作内容形成社区 |

## 二、项目背景与历史

2. 调研样本量小，后续需要更充分的市场验证

## 三、需求阶段

### 议题1：需求来源与产品定位

**PM-老张**：需求是怎么来的？谁提出来的？

**产品-小王**：主要是我根据市场调研和竞品分析提出的。

**PM-老张**：用户验证了吗？需求优先级怎么定的？

**产品-小王**：用户验证确实不够充分。优先级是按功能重要性排的，核心功能优先。

**PM-老张**：什么叫"功能重要性"？有量化标准吗？

**产品-小王**：...没有量化，主要是凭经验判断。

**PM-老张**：凭经验？那万一判断错了呢？

**PMO-小李**：**方案一：MoSCoW优先级法。** Must have（必须有）、Should have（应该有）、Could have（可以有）、Won't have（不会有）。优点是清晰明确，缺点是需要充分讨论。

**PMO-小李**：**方案二：Kano模型。** 基本型、期望型、兴奋型需求分类。优点是考虑用户满意度，缺点是调研成本高。

**PMO-小李**：**方案三：价值/成本矩阵。** 高价值低成本优先，低价值高成本后置。优点是量化决策，缺点是价值评估主观。

**PM-老张**：下次用MoSCoW法，这次先记下来。

**【决议】**
1. 需求来源：产品经理提出，用户验证不充分
2. 优先级判断主观，缺乏量化标准
3. 后续使用MoSCoW优先级法

### 议题2：用户画像构建

**前端-小陈**：用户画像是什么？目标用户是谁？

**产品-小王**：10-40岁，喜欢故事创作和卡牌收集的用户，主要面向海外市场。

**PM-老张**：太笼统了。10岁和40岁的需求能一样吗？海外哪个市场？美国还是欧洲？

**PM-老张**：翻译质量怎么样？谁翻译的？

**产品-小王**：AI翻译+人工校对。

**PM-老张**：AI翻译？文化差异考虑了吗？比如"职场风云"在美国可能不适用。

**产品-小王**：确实有这个问题，有些书籍类型在海外市场接受度可能不高。

**后端-小刘**：**方案一：完全本地化。** 每个市场定制内容，翻译+文化适配。优点是体验好，缺点是成本高。

**后端-小刘**：**方案二：部分本地化。** 核心内容本地化，其他内容机器翻译。优点是成本低，缺点是体验一般。

**后端-小刘**：**方案三：用户自选语言。** 用户选择语言后，只展示该语言的内容。优点是简单，缺点是内容受限。

**【决议】**
1. 当前国际化：中英文支持，AI翻译

## 四、设计阶段

### 议题1：技术选型决策

**PM-老张**：为什么选Cloudflare全家桶？

**后端-小刘**：Serverless架构，按需付费，成本低。而且全球CDN，性能好。

**PM-老张**：有没有评估其他方案？

**后端-小刘**：评估了Vercel、AWS Lambda、阿里云函数计算。

**PM-老张**：为什么选Cloudflare？

**后端-小刘**：**方案一：Cloudflare全家桶。** Pages + Functions + D1。优点是一体化、成本低、全球CDN。缺点是D1功能有限、生态不如AWS成熟。

**后端-小刘**：**方案二：Vercel + Supabase。** Vercel托管 + Supabase数据库。优点是开发体验好、数据库功能强。缺点是成本可能更高。

**后端-小刘**：**方案三：AWS Lambda + RDS。** AWS全家桶。优点是功能最全、生态最成熟。缺点是复杂度高、成本不可控。

**PM-老张**：选Cloudflare的原因是什么？

**后端-小刘**：成本最低，而且项目规模小，D1功能够用。

**PM-老张**：D1有限制，流量上来怎么办？

**运维-老孙**：D1免费版限制：5GB存储、500万行读取/天。流量上来需要付费版。

**【决议】**
1. 技术选型：Cloudflare全家桶
2. 选型原因：成本最低，功能够用
3. 风险：D1有限制，流量上来需要升级

### 议题2：架构设计思路

**前端-小陈**：前端架构是什么？

**前端-小陈**：原生HTML/CSS/JS，没有使用框架。

**PM-老张**：为什么不用框架？

**前端-小陈**：时间紧，而且项目规模小，原生JS够用。

**PM-老张**：现在够用，以后呢？代码复用性怎么办？

**前端-小陈**：**方案一：保持现状。** 原生JS，模块化开发。优点是轻量，缺点是开发效率低。

**前端-小陈**：**方案二：引入Vue。** 渐进式引入，新功能用Vue。优点是开发效率高，缺点是新旧代码混合。

**前端-小陈**：**方案三：引入Web Components。** 原生组件化，无框架依赖。优点是标准、轻量，缺点是生态不成熟。

**PM-老张**：先保持现状，记录技术债务，后续评估重构。

**【决议】**
1. 前端架构：原生HTML/CSS/JS
2. 技术债务：代码复用性差
3. 后续评估引入框架或Web Components

### 议题3：数据库设计经验

**后端-小刘**：数据库设计遵循了什么原则？

**后端-小刘**：ER关系清晰，用户→书籍→角色/卡牌/章节，每张表都有user_id实现数据隔离。

**PM-老张**：索引设计了吗？

**后端-小刘**：主要查询字段都加了索引。

**PM-老张**：有没有性能测试？

**测试-小赵**：没有做专门的数据库性能测试。

**后端-小刘**：**方案一：添加性能测试。** 编写数据库性能测试用例，模拟高并发场景。优点是发现问题早，缺点是需要时间。

**后端-小刘**：**方案二：监控慢查询。** 上线后监控慢查询日志，发现问题再优化。优点是省时间，缺点是可能影响用户。

**后端-小刘**：**方案三：数据库设计评审。** 上线前做数据库设计评审，专家把关。优点是质量高，缺点是需要专家资源。

**PM-老张**：方案一+方案二组合。上线前做基础性能测试，上线后监控慢查询。

**【决议】**
1. 数据库设计：ER关系清晰，数据隔离
2. 缺少性能测试
3. 上线前做基础性能测试，上线后监控慢查询

### 议题4：UI设计差异化

**产品-小王**：4种书籍类型有差异化UI设计，这个做得怎么样？

**前端-小陈**：每种类型有不同的主题色和风格，儿童冒险明亮活泼，魔幻传说神秘奇幻，都市言情温馨浪漫，职场风云专业简洁。

**PM-老张**：设计稿谁出的？

**产品-小王**：我出的设计思路，前端实现。

**PM-老张**：有设计规范吗？

**前端-小陈**：没有统一的设计规范，每个类型独立设计。

**PM-老张**：没有规范？那怎么保证一致性？

**前端-小陈**：**方案一：建立设计系统。** 统一的设计规范、组件库、设计Token。优点是一致性好，缺点是工作量大。

**前端-小陈**：**方案二：设计模板化。** 每种类型一个设计模板，新页面基于模板。优点是简单，缺点是灵活性差。

**前端-小陈**：**方案三：设计评审机制。** 每个新设计需要评审，确保一致性。优点是灵活，缺点是依赖人工。

**PM-老张**：方案二+方案三组合。设计模板化+设计评审。

**【决议】**
1. UI差异化设计完成，但缺少统一规范
2. 后续建立设计模板+设计评审机制
3. 考虑建立设计系统

## 二十、会议总结

### 核心共识

1. **技术目标达成，业务目标待验证**：项目已上线，核心功能可用，但用户留存率低，需要持续优化。

2. **用户验证不足是最大教训**：需求凭经验判断，没有充分验证，导致产品做出来了但用户不买账。

3. **技术债务需要管理**：原生JS开发、语音服务预留未用等问题需要记录和排期处理。

4. **风险意识需要加强**：AI服务不稳定、成本可能失控、合规风险等问题需要提前应对。

5. **效率提升空间大**：研发效率、测试效率、需求效率都有提升空间，可以通过AI辅助、自动化、流程优化来改善。

### 后续重点

1. **用户优先**：先解决留存问题，再优化技术。
2. **成本控制**：监控用量、限流缓存、设定成本上限。
3. **合规完善**：用户协议、隐私政策、未成年人保护。
4. **效率提升**：AI辅助、自动化、流程优化。
