## 第一章：项目管理经验

### 1.1 项目进度管理方法

**核心方法**：里程碑管理 + 迭代开发

**里程碑设置原则**：

| 原则 | 说明 |
|------|------|
| 可量化 | 每个里程碑有明确的交付物 |
| 可达成 | 里程碑目标合理可达 |
| 有时限 | 每个里程碑有明确的截止日期 |
| 有优先级 | 里程碑按重要性排序 |

**如何应用到新项目**：

| 步骤 | 说明 |
|------|------|
| 1. 识别关键节点 | 列出项目的关键节点 |
| 2. 设置里程碑 | 每个节点设置里程碑 |
| 3. 定义交付物 | 明确每个里程碑的交付物 |
| 4. 分配资源 | 为每个里程碑分配资源 |
| 5. 跟踪进度 | 定期跟踪里程碑进度 |

### 1.2 风险管理经验

| 合规风险 | 法规研究、专家咨询 | 合规检查清单 |

**如何应用到新项目**：

| 步骤 | 说明 |
|------|------|

| 3. 风险应对 | 制定应对策略 |

| 5. 风险复盘 | 项目结束后复盘风险应对效果 |

### 1.3 资源协调经验

**资源协调原则**：

### 本章小结

**核心方法论**：
1. 里程碑管理：可量化、可达成、有时限、有优先级

## 第三章：技术选型经验

### 3.1 Serverless架构选型思考

**选型对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Cloudflare全家桶 | 成本低、全球CDN、一体化 | D1功能有限、生态不成熟 | 小型项目、成本敏感 |
| Vercel + Supabase | 开发体验好、数据库功能强 | 成本可能更高 | 中型项目、重视开发体验 |
| AWS全家桶 | 功能最全、生态最成熟 | 复杂度高、成本不可控 | 大型项目、需要完整生态 |

**如何应用到新项目**：

| 步骤 | 说明 |
|------|------|
| 1. 明确需求 | 明确项目的技术需求 |
| 2. 列出候选方案 | 列出可能的技术方案 |
| 3. 对比评估 | 从成本、功能、生态等维度对比 |
| 4. POC验证 | 对候选方案做POC验证 |
| 5. 做出决策 | 综合评估后做出决策 |

**优缺点分析**：

| 中型项目 | 可以使用，注意D1限制 |
| 大型项目 | 谨慎使用，评估功能是否满足 |

## 第六章：踩坑经验汇总

### 6.1 数据隔离问题

**问题描述**：多用户系统中，查询不包含user_id过滤，导致用户A看到用户B的数据。

**根本原因**：开发时忘记添加user_id过滤条件。

**解决方案**：所有涉及用户数据的查询，必须在WHERE条件中包含user_id过滤。

**如何应用到新项目**：

| 步骤 | 说明 |
|------|------|
| 1. 识别用户数据 | 列出所有需要隔离的数据表 |
| 2. 添加user_id字段 | 确保每张表都有user_id字段 |
| 3. 修改查询语句 | 所有查询添加WHERE user_id = ? |
| 4. 添加自动化检查 | 编写测试用例验证数据隔离 |
| 5. 代码审查要点 | PR审查时检查是否遗漏user_id过滤 |

### 6.2 DOM空值访问问题

**问题描述**：访问DOM元素时未检查是否存在，导致JavaScript错误。

**根本原因**：假设DOM元素一定存在，未做防御性编程。

**解决方案**：访问DOM元素前先检查是否存在。

**如何应用到新项目**：

| 步骤 | 说明 |
|------|------|
| 1. 识别DOM访问 | 找出所有DOM访问代码 |
| 2. 添加空值检查 | 在访问前添加空值检查 |
| 3. 编写防御性代码 | 使用可选链操作符等 |
| 4. 添加错误处理 | 添加try-catch错误处理 |
| 5. 测试边界情况 | 测试元素不存在的情况 |

### 6.3 字段命名一致性问题

**问题描述**：前端和后端字段命名不一致，导致数据绑定错误。

**根本原因**：前后端开发人员沟通不足，各自命名。

**解决方案**：统一字段命名规范，前后端使用相同命名。

**如何应用到新项目**：

| 步骤 | 说明 |
|------|------|
| 1. 制定命名规范 | 制定统一的字段命名规范 |

| 4. 自动化检查 | 使用工具自动检查一致性 |
| 5. 文档化 | 文档化字段命名规范 |

### 6.4 数据库迁移问题

**问题描述**：数据库迁移脚本执行顺序错误，导致数据丢失。

**根本原因**：迁移脚本没有版本控制，执行顺序混乱。

**解决方案**：迁移脚本按版本号命名，按顺序执行。

**如何应用到新项目**：

| 步骤 | 说明 |
|------|------|
