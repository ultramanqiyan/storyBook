## 一、项目概述

### 1.1 项目目标

创建一个卡牌互动小说书籍网站，用户可以通过收集卡牌、选择角色和情节来创作和阅读互动小说故事。

### 1.2 核心价值

| 价值点 | 描述 |
|--------|------|
| 互动阅读 | 用户可以选择角色和情节参与故事创作 |
| 卡牌收集 | 通过解谜获得新卡牌，丰富创作元素 |
| 多样化体验 | 4种书籍类型，每种有独特的视觉风格 |
| AI辅助创作 | 章节内容由AI生成，每次体验独一无二 |

### 1.3 目标用户

- 喜欢故事创作的读者
- 喜欢卡牌收集的玩家
- 10-40岁，对互动小说感兴趣的用户

## 三、核心功能

### 3.1 用户系统

#### 3.1.1 注册与登录

| 功能 | 描述 |
|------|------|
| 注册方式 | 邮箱 + 密码 |
| 密码存储 | bcrypt加密 |
| 会话管理 | userId存储在localStorage |
| 验证方式 | 第一阶段不验证邮箱，仅收集 |

#### 3.1.2 用户权限

| 场景 | 未登录用户 | 登录用户 |
|------|-----------|----------|
| 查看系统预设书籍 | ✅ 可以 | ✅ 可以 |
| 阅读章节内容 | ✅ 可以 | ✅ 可以 |
| 创建新书籍 | ❌ 跳转登录 | ✅ 可以 |
| 添加章节 | ❌ 跳转登录 | ✅ 可以 |

### 3.2 书籍系统

#### 3.2.1 书籍类型

| 类型 | 描述 | 特点 |
|------|------|------|
| 儿童冒险 | 适合儿童的故事 | 明亮、卡通 |
| 魔幻传说 | 魔法、奇幻主题 | 紫色、金色、星空 |
| 都市言情 | 现代都市爱情 | 粉色、浪漫 |
| 职场风云 | 职场故事 | 蓝色、商务 |

#### 3.2.2 预设书籍

- 每个类型有1本预设书籍
- 每本预设书籍有4个章节
- 每个章节约300字
- 未登录用户可以查看完整内容

#### 3.2.3 书籍创建

创建书籍时需要填写：
1. 书籍名称（必填）
2. 书籍类型（必选）
3. 主角信息（必填）：
   - 名称
   - 性格（预设+自定义）
   - 说话方式（预设+自定义）
   - 角色类型（预设+自定义）
   - 头像（预设）
4. 配角信息（0-3个）：
   - 名称
   - 头像
   - 性格
   - 说话方式
   - 角色类型
   - 与主角亲密度（中立/友好/敌对）
   - 与主角关系

### 3.3 卡牌系统

#### 3.3.1 卡牌类型

| 大类 | 子类 | 上限 | 说明 |
|------|------|------|------|
| 角色卡牌 | - | 8张 | 1个主角 + 0-7个配角 |
| 情节卡牌 | 天气 | 8张 | 每种类型独立上限 |
| 情节卡牌 | 地形 | 8张 | - |
| 情节卡牌 | 冒险类型 | 8张 | - |
| 情节卡牌 | 装备 | 8张 | - |

#### 3.3.2 卡牌获取

1. **创建书籍时**：自动创建主角和配角角色卡牌
2. **解谜成功后**：随机掉落1张情节卡牌

#### 3.3.3 卡牌上限检查

- 解谜成功获得卡牌时检查
- 添加新章节前检查
- 超过上限时弹窗要求丢弃卡牌
- 不丢弃无法继续操作

### 3.4 章节系统

#### 3.4.1 添加章节流程

1. 点击"添加章节"跳转到故事导演页
2. 选择主角（必选）
3. 选择情节卡牌（每种类型各选1张）：
   - 天气卡牌
   - 地形卡牌
   - 冒险类型卡牌
   - 装备卡牌
4. 点击"开始拍摄"
5. 系统构造提示词（假数据）
6. 返回新章节内容
7. 跳转到书籍详情页

#### 3.4.2 章节内容

- 固定300字左右
- 纯标题格式（不含"第X章"）
- 包含谜题

### 3.5 谜题系统

#### 3.5.1 谜题配置

| 项目 | 配置 |
|------|------|
| 每个章节 | 固定有1个谜题 |
| 尝试次数 | 最多3次 |
| 成功奖励 | 掉落1张情节卡牌 |
| 失败无奖励 | 不掉落卡牌 |

#### 3.5.2 谜题类型

- 文字谜题（猜成语、字谜）
- 逻辑谜题（数字规律、推理）
- 互动谜题（选择题）

  success: true,

  error: null

### 6.2 API列表

| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/users` | POST | 用户注册/登录 |
| `/api/users` | GET | 获取用户信息 |

## 十一、预防措施（来自历史项目经验）

### 11.1 必避Bug

| Bug | 预防措施 |
|-----|----------|
| 登录逻辑 | 先查询用户是否存在，再决定创建或登录 |
| 数据隔离 | 所有查询必须包含user_id过滤 |
| DOM空值 | 所有DOM操作前检查元素是否存在 |
| CSS点击区域 | 交互元素z-index > 100 |
| 数据类型比较 | SQLite布尔字段使用0/1或truthy判断 |
| 3D transform | 使用transform时重新设置z-index |
| 接口字段名 | 定义API文档，统一字段命名 |
| 数据库迁移 | 迁移文件使用版本号命名 |
| D1绑定 | 本地开发不使用--d1参数 |

### 11.2 开发规范

1. **防御性编程**：参数校验、空值检查、错误处理
2. **统一响应格式**：{ success, data, error }
3. **参数化查询**：防止SQL注入
4. **日志记录**：关键操作记录日志，便于问题定位
